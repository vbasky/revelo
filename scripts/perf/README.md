# Performance Comparison Tools

These scripts are local developer tools for comparing Revelo with reference
tools such as MediaInfo and ffprobe. They are not part of normal Revelo
execution and they do not commit benchmark outputs.

The pipeline supports both generated sparse fixtures and real local media from
one manifest. Private file paths may appear in local manifests, but generated
results omit paths by default.

Process-level timings come from `hyperfine` only. The Python runner is an
orchestrator for manifests, fixtures, JSON sanitization and table rendering; it
does not use Python wall-clock timing as a fallback.

Required tools for the full comparison:

- `hyperfine`
- `mediainfo`
- `ffprobe`

Use `--skip-mediainfo` or `--skip-ffprobe` only for local smoke tests where the
reference tool is intentionally absent.

## Entry Points

Run the full comparison from a local manifest:

```sh
just bench-compare manifest=scripts/perf/manifest.example.json
```

Run the full PR evidence pipeline with a shared run id:

```sh
just bench-evidence manifest=path/to/local-manifest.json
```

Include the optional WASM probe in the same run directory:

```sh
just bench-evidence-wasm manifest=path/to/local-manifest.json
```

Write only sanitized JSON:

```sh
just bench-compare-json manifest=path/to/local-manifest.json
```

Regenerate the standalone HTML table from existing JSON:

```sh
just bench-compare-table \
  results=target/perf-investigation/<run-id>/results.json
```

Regenerate the standalone HTML table with oracle-parity status:

```sh
just bench-evidence-table \
  results=target/perf-investigation/<run-id>/results.json \
  oracle=target/perf-investigation/<run-id>/oracle-parity.json
```

Regenerate the PNG from existing HTML when Playwright is available:

```sh
just bench-compare-png \
  html=target/perf-investigation/<run-id>/benchmark-table.html
```

Run a targeted oracle-parity pass against MediaInfo JSON:

```sh
just bench-oracle manifest=path/to/local-manifest.json
```

Run a WASM parse probe:

```sh
just bench-wasm manifest=path/to/local-manifest.json
```

`bench-compare` builds the release CLI and the `perf_probe` example before
running measurements. It fails if `hyperfine` is not installed.

`bench-evidence` runs the comparison, oracle parity check and table rendering
through `run_benchmark_evidence.py`, using one shared `run_id`, fixture
directory and output directory. The rendered table includes an `Oracle` column
when `oracle-parity.json` is available. It writes:

```text
target/perf-investigation/<run-id>/
  results.json
  benchmark-table.html
  benchmark-table.png        # optional, when Playwright capture is available
  oracle-parity.json
  evidence-summary.json
```

`bench-evidence-wasm` adds `wasm-probe.json` to the same directory. Use it only
when `wasm-pack`, Node and the `wasm32-unknown-unknown` Rust target are
available.

## Outputs

The default output layout is:

```text
target/perf-investigation/<run-id>/
  results.json
  benchmark-table.html
  benchmark-table.png        # optional

target/perf-fixtures/<run-id>/
  <generated synthetic media>
```

These files are generated artifacts and should not be committed.

`results.json` records the measurement backend:

```json
{
  "measurement_backend": {
    "name": "hyperfine",
    "required": true
  }
}
```

## Manifest Modes

The manifest schema is `revelo_perf_manifest_v2`. Each case must contain
exactly one source:

- `synthetic`: generated by `scripts/perf/generate_fixtures.py`;
- `path`: real local media, usually from a private or public corpus checkout.

Generated-only manifest:

```json
{
  "schema": "revelo_perf_manifest_v2",
  "settings": { "warmups": 2, "runs": 10 },
  "cases": [
    {
      "id": "synthetic-mp4-snv2-tail-389m",
      "label": "MP4 / SNV2 / moov tail / 389 MiB",
      "class": "synthetic",
      "container": "MP4",
      "codec": "H.264",
      "layout": "SNV2 brand, moov tail, sparse mdat",
      "source": "generated",
      "synthetic": {
        "kind": "mp4_snv2_tail",
        "size_bytes": 407896064
      }
    }
  ]
}
```

Real-only manifest:

```json
{
  "schema": "revelo_perf_manifest_v2",
  "settings": { "warmups": 2, "runs": 10 },
  "cases": [
    {
      "id": "real-mp4-avc-local",
      "label": "MP4 / AVC / real local sample",
      "class": "real",
      "container": "MP4",
      "codec": "H.264",
      "layout": "real media",
      "source": "local private or public corpus label",
      "path": "local-media/not-committed/real-mp4-avc.mp4"
    }
  ]
}
```

Mixed manifests use both case forms in the same `cases` array.

The manifest can also compare multiple Revelo binaries in the same run. Omit
`revelo_versions` to measure the current `target/release/revelo` as `Revelo`.
When `revelo_versions` is present, each entry becomes one timing column. Paths
may be absolute or relative to the repository root and should point at local
binaries prepared outside git, for example:

```json
{
  "schema": "revelo_perf_manifest_v2",
  "revelo_versions": [
    { "id": "revelo_0_4_6", "label": "Revelo 0.4.6", "path": "target/perf-tools/revelo-0.4.6" },
    { "id": "revelo_0_5_0", "label": "Revelo 0.5.0", "path": "target/perf-tools/revelo-0.5.0" },
    { "id": "revelo_0_5_1", "label": "Revelo 0.5.1", "path": "target/perf-tools/revelo-0.5.1" },
    { "id": "revelo_branch", "label": "Revelo branch", "path": "target/release/revelo" }
  ],
  "cases": [
    {
      "id": "synthetic-mp4-snv2-tail-389m",
      "label": "MP4 / SNV2 / moov tail / 389 MiB",
      "class": "synthetic",
      "container": "MP4",
      "synthetic": { "kind": "mp4_snv2_tail", "size_bytes": 407896064 }
    }
  ]
}
```

## Table Config

`table.config.example.json` controls presentation:

- template path;
- stylesheet path;
- PNG capture element id;
- optional visible tool columns;
- display names for columns;
- row sections;
- row group order;
- latency color thresholds;
- optional baseline metadata.

The renderer produces a standalone table, not a dashboard. When
`--oracle-results` is provided, it adds one compact status column per case:
`pass`, `diff`, `fail` or `n/a`. The PNG capture targets only the
`#benchmark-table-capture` element so screenshots do not need manual cropping.

The default template lives at `scripts/perf/templates/benchmark-table.html` and
the default stylesheet lives at `scripts/perf/styles/benchmark-table.css`.
The template uses simple `{{ name }}` placeholders implemented with the Python
standard library; Jinja2 is not required.

## Oracle Parity

`run_oracle_parity.py` compares Revelo `--json` output with
`mediainfo --Output=JSON` on the same manifest cases. It is intended for the
cap/truncation question: use real risky files plus required fields that should
survive bounded metadata reads.

The default entrypoint is:

```sh
just bench-oracle manifest=path/to/local-manifest.json \
  config=scripts/perf/oracle.config.example.json
```

The config schema is `revelo_oracle_parity_config_v1`:

- `ignore_fields`: volatile or path-derived MediaInfo fields to ignore;
- `required_fields`: `{ "track_type": "...", "field": "..." }` pairs that
  must match MediaInfo when MediaInfo reports them;
- `case_overrides`: optional per-case additions keyed by manifest `id`.

The output is `target/perf-investigation/<run-id>/oracle-parity.json`. It omits
input paths and keeps only labels, sizes, required-field failures, common-field
value differences and missing tracks. Use `--fail-on-diff` when a strict local
gate is desired.

## WASM Probe

`run_wasm_probe.py` measures `revelo-wasm` through Node after the file has
already been read into memory. This is deliberately not the same measurement as
the CLI benchmark: it excludes mmap, page faults, stdout and filesystem read
time, and isolates the `wasm_bindgen` `parse(Uint8Array)` path.

The default entrypoint is:

```sh
just bench-wasm manifest=path/to/local-manifest.json
```

Requirements:

- `wasm-pack`;
- `node`;
- Rust target `wasm32-unknown-unknown`.

The output is `target/perf-investigation/<run-id>/wasm-probe.json`, with
sanitized case labels and per-run parse samples. This answers the
sequential-detect/WASM question separately from desktop CLI timing.

## Supported Synthetic Fixture Kinds

Current fixture kinds:

- `mp4_moov_front`
- `mp4_moov_tail`
- `mp4_snv2_tail`
- `mov_moov_tail`
- `fragmented_mp4`
- `webm_sparse`
- `mkv_sparse`
- `wav_list_id3_data`
- `bwf_data`
- `rf64_ds64_data`
- `aiff_ssnd`
- `aifc_ssnd`
- `flac_large`
- `mp3_id3_large`
- `ogg_large`
- `mpeg_ts_large`
- `mpeg_ps_large`
- `avi_large`

The generated files are parser-oriented sparse fixtures. They are useful for
bounded-access and cross-tool timing checks, but they do not replace real media
corpus coverage.

## Privacy Contract

Committed files may include public labels, sizes, tool versions and aggregate
timings. Do not commit:

- real media;
- generated fixtures;
- local manifests with private paths;
- raw logs;
- generated JSON, HTML or PNG outputs.

`perf_probe` also omits the input path from JSON by default. Use
`--include-path` only for private local debugging output.
